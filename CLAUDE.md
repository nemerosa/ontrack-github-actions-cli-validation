## Commands

```bash
npm run lint    # ESLint (flat config, v10)
npm run build   # esbuild bundle to dist/index.js + esbuild-plugin-license writes dist/licenses.txt
npm test        # Jest unit tests (mocked @actions/* deps)
npm run all     # lint + build + test
```

`npm run build` runs `node build.js`, which invokes esbuild with `esbuild-plugin-license` to produce `dist/index.js` and license attribution. ncc is no longer used.

## Architecture

This is a JavaScript GitHub Action that creates a validation run in [Yontrack](https://github.com/nemerosa/ontrack-cli) for an existing build. It takes a `validation` stamp name and an optional explicit `status` (PASSED, FAILED, etc.), with optional `type`+`flags` for typed validation data, and invokes the Yontrack CLI's `validate` subcommand.

**Project / branch / build resolution:**
- Each can be passed explicitly as an action input
- Otherwise falls back to the corresponding `YONTRACK_PROJECT_NAME` / `YONTRACK_BRANCH_NAME` / `YONTRACK_BUILD_NAME` env var (set by a prior `cli-config` step)
- For project/branch only: further fallback to `github.context.repo.repo` and parsing `github.context.ref` if env vars are unset
- `build` has no further fallback â€” must be provided either explicitly or via the env var

**Key files:**
- `index.js` â€” source. A top-level IIFE loads pure-ESM `@actions/*` packages via dynamic import and calls `runAction({core, exec, github})`.
- `dist/index.js` â€” bundled output (esbuild, CJS). The runner executes this.
- `dist/licenses.txt` â€” license attributions, generated by `esbuild-plugin-license`.
- `action.yml` â€” action metadata: inputs (project, branch, build, validation, status, type, flags, executable, logging), `runs.using: node24`, `runs.main: dist/index.js`.
- `index.test.js` â€” Jest unit tests with hand-rolled mock objects.
- `build.js` â€” esbuild driver with `esbuild-plugin-license`.
- `.yontrack/ci.yaml` â€” cli-config target config, used by the previous step in CI workflows.

**Why dependency injection:** All `@actions/*` packages at the latest majors are pure ESM. Top-level CJS imports no longer work. The IIFE loads ESM deps once at startup and passes them into `runAction`.

**History note:** Prior to this overhaul, the action implemented an "API polling" mode that took a `step-name` input and queried GitHub Actions for that step's completion status. The maintainer's commit `9e9af49 feat: explicit status` (Nov 2025) updated `action.yml` to advertise an explicit-status interface but never updated `index.js`. This upgrade completes that migration â€” `step-name` and the polling logic are removed; `status` is now read from inputs as `action.yml` describes.

## Release process

Releases are automated via `semantic-release` (`.releaserc`). Commit messages follow Angular convention. Every conventional-commit type triggers at least a patch release; `feat` is minor; `refactor` and any commit with a `BREAKING CHANGE:` footer is major.

The CI workflow on `main`:
1. Runs `npm run lint && npm run build && npm test`.
2. Configures Yontrack via `cli-config` (which sets `YONTRACK_*` env vars).
3. Runs `npx semantic-release` â€” produces `vX.Y.Z` GitHub release; `@semantic-release/git` commits the regenerated `dist/`, `CHANGELOG.md`, `package.json`, `package-lock.json`, and `action.yml`.
4. Force-updates floating `vX` and `vX.Y` tags to point at the new release SHA.
